PENTEST REPORT

Comprehensive Security Assessment

Project Overview

Summary
This project took place within the framework of the Application Security course at Thomas More. The assignment was geared towards thoroughly assessing the security of a web application and compiling a detailed report of the discovered vulnerabilities. To begin, I established a controlled testing environment by configuring a web application on a virtual Ubuntu Server, accessed through my Kali Linux virtual machine. Our instructors introduced a diverse range of undisclosed vulnerabilities, each tailored to challenge students individually.

Utilizing Burp Suite's robust proxy and repeater tools, I methodically examined the application's functionality and susceptibility to potential threats. Throughout the penetration testing process, I diligently explored various facets of the application to identify and analyze potential vulnerabilities.

It's worth noting that this pentest project encompassed a comprehensive evaluation of vulnerabilities across topics covered in the course curriculum. These topics served as the framework for our analysis, guiding our efforts towards a thorough assessment of the application's security posture.

Topics for testing

  • SQL Injection
  • Command Injection
  • Directory Traversal
  • Information Disclosure
  • Authentication
  • Buffer Overflows & Remote Access
  • Server-Side Request Forgery (SSRF)
  • Cross-Site Scripting (XSS)
  • Cross-Origin Resource Sharing (CORS)
  • API Testing
  • File Upload Vulnerabilities
  • Cross-Site Request Forgery (CSRF)

What did I learn?
Throughout the project, I developed technical skills in web application security testing, particularly using tools like Burp Suite. I gained hands-on experience with virtual environments such as Ubuntu Server and Kali Linux. Employing systematic testing methodologies, I honed my problem-solving and vulnerability analysis skills. Moving forward, I recognize the importance of continuous learning to stay ahead in the dynamic field of cybersecurity.

AWS/GITLAB APP HOSTING

Cloud-Native CI/CD Pipeline

Project Overview

Summary
For this group project in the course Project 4.0, my team and I had to build a weather application designed specifically for farmers. The app allowed them to register their crops, track weather conditions, and submit insurance claims in case of damage. Our team was made up of students from three different specializations: App Development, Artificial Intelligence, and Cloud & Cybersecurity. I was the only student representing the Cloud & Cybersecurity track.

My Responsibility
My main responsibility was setting up the cloud infrastructure and ensuring the project was secure and automated. I created multiple GitLab repositories for the developers and implemented CI/CD pipelines that built Docker images of their code and deployed them automatically to AWS. This was all done using the AWS CLI directly inside the GitLab pipelines. I also made adjustments in the developers‚Äô code when needed, particularly to ensure Docker containers communicated properly and everything ran smoothly together. This required me to keep a bird‚Äôs-eye view of the project so that all parts integrated seamlessly.

Infrastructure as Code
The infrastructure itself was fully automated. From setting up VPCs, subnets, and EC2 instances, to configuring security groups and Elastic IPs, everything was handled through code. To give the team flexibility, I included manual triggers in the pipelines so we could choose when to run specific stages (e.g., setup, testing, cleanup). I was also responsible for managing and submitting all project documentation, ensuring everything met the school‚Äôs formatting and content guidelines.

Pipelines
We structured our CI/CD around three main repositories: Frontend, Backend, and Infrastructure, each with its own .gitlab-ci.yml pipeline file. All pipelines included the directive when: manual, allowing us to decide which stages to run and when.

Infrastructure Pipeline
The infrastructure pipeline contained five main stages:
‚Äì Setup Infrastructure Environment
‚Äì Setup EC2 Instance
‚Äì Test Infrastructure
‚Äì Cleanup EC2 Instance
‚Äì Cleanup Infrastructure Environment

In the Setup Infrastructure Environment stage, core resources were created, such as:
‚Äì ECR repositories (to store Docker images built by the other pipelines)
‚Äì VPC (10.0.0.0/16) with public and private subnets, NAT gateway, and route tables
‚Äì Security groups allowing HTTP(80), HTTPS(443), SSH(22), Angular(4200), FastAPI(8000), and MongoDB(27017)
‚Äì S3 bucket to store artifacts like docker-compose.yml, SSH key pairs, and metadata files

During the Setup EC2 Instance stage, the pipeline provisioned an EC2 server with Elastic IPs, extended its root disk to 16 GB, mounted additional EBS volumes for database data, installed Docker, and pulled down configuration files from S3. This allowed the containers to run persistently with data mounted to volumes.

The Test Infrastructure stage validated that resources like subnets, security groups, and Elastic IPs were properly set up. Finally, the Cleanup stages destroyed the resources in reverse order to ensure no unnecessary costs.

Frontend & Backend Pipelines
The frontend and backend pipelines were similar, with extra dependencies and quality stages for the frontend to run npm tests. Both pipelines included:
‚Äì Build: Installed Docker Compose, retrieved SSL certificates from S3, built Docker images, and pushed them to AWS ECR.
‚Äì Deploy: Retrieved artifacts and SSH keys, connected to the EC2 instance, stopped old containers, pulled the new images from ECR, and restarted them.

Docker Setup
The infrastructure repository contained the docker-compose.yml, which defined three main containers: frontend (Angular on port 4200), backend (FastAPI on port 8000), and database (MongoDB on port 27017). The backend‚Äôs Dockerfile used Python 3.9 with FastAPI and pymongo, while the frontend used a multi-stage build (Node.js to compile Angular, then served by Nginx).

Outcome
In the end, the project was a success. The developers could focus on their part without worrying about infrastructure issues, and I learned a lot about cloud automation, infrastructure as code, teamwork, and cross-disciplinary collaboration. I also experienced firsthand how valuable it is to take ownership of the entire infrastructure and make sure every component works as a unified system.

What did I learn?
This project taught me what it really means to take ownership of infrastructure in a team setting. I didn‚Äôt just automate a few tasks, I built and managed the entire cloud environment from scratch, and I had to think several steps ahead to keep everything working together. It was the first time I truly experienced how valuable Infrastructure as Code is, and how much time and frustration it saves when done right.

I also learned how important it is to communicate clearly with teammates from different backgrounds. The developers weren‚Äôt always familiar with cloud tooling or containerization, so I had to explain certain decisions in a way that made sense to them. That improved my ability to translate complex technical topics into practical insights for others.

Something I didn‚Äôt expect going in was how often I had to troubleshoot problems that weren‚Äôt technically ‚Äúmine.‚Äù But instead of saying ‚Äúthat‚Äôs not my responsibility,‚Äù I took initiative and made sure everything worked as one system, even if that meant diving into unfamiliar code. That mindset shift‚Äîfrom doing my own part to actively owning the project‚Äôs stability‚Äîis probably the most valuable thing I took away.

GPS PUPIL TRACKER

IoT Location Tracking System

Project Overview

Overview
This project took place within the framework of the Systems Engineering & Analysis (based on Prince2) course at Thomas More Geel. It involved collaboration with 5 of my classmates. We had to create a proof of concept with the aim of developing an IoT device and application that could track children for a teacher or supervisor. The user (teacher) can define a zone where the children must stay within, and in case a child leaves the zone, both the teacher and the child should receive a notification.

Additionally, emergency buttons had to be provided for both the teacher and the child. Furthermore, everything had to be safe and user-friendly. We accomplished this project by utilizing an ESP32, a Firebase database, and a hosted site that interacts with the database. The choice of this project stemmed from the desire to explore and understand the Scrum process model. It is important to note that our team successfully managed to implement all the must-haves and should-haves, as requested by the client.

My Contributions
My focus was on perfecting and implementing the JavaScript code that seamlessly interacts with the back-end. This task involved not only writing structured and efficient code but also implementing visual enhancements in the User Interface to optimize the overall user experience.

What did I learn?
I gained valuable experience working in an agile (Scrum) team environment. This project allowed me to enhance my soft skills such as communication, teamwork, and adaptability. Furthermore, I gained a deeper understanding of how complex project components are developed, interact with each other, and ultimately come to fruition. My focus on perfecting and implementing JavaScript code for the project's user interface not only honed my technical skills but also emphasized the importance of user-centric design in creating a safe and user-friendly solution.
